# 测试版面 Spec

## 目标

在求职准备专区新增一个名为「测试版面」的页面，把 `06_面试题精准解答.md` 中的全部 128 道题导入到站内题库。页面同时支持题库总览、筛选检索、题目详情阅读，以及随机抽题自测。

当前源文件结构：

- 7 个主类：RAG 系统开发、Agent 智能体开发、Prompt 工程、框架与工具、微调、应用部署与工程化、场景设计题。
- 30 个小节。
- 128 道题。
- 每题结构基本固定：核心概念、原理机制、实践要点、常见追问、面试话术。

题量分布：

| 主类 | 题数 |

解析规则：

- `## ` 解析为主类，排除「面试题来源与参考资料」。
- `### ` 解析为小节。
- `#### Q\d+:` 解析为题目。
- `**核心概念：**`、`**原理机制：**`、`**实践要点：**`、`**常见追问：**`、`**面试话术：**` 作为标准 block。
- 保留代码块、表格、列表和引用。
- 如果某题缺少某个标准 block，输出空字符串，不阻塞构建。
- 构建结束必须打印：题目总数、主类分布、小节数量、缺失 block 列表。

## 页面信息架构

### 顶部

- 标题：「测试版面」
- 副标题：`128 道大模型应用开发岗面试题 · 总览 / 检索 / 随机自测`
- 指标条：
  - 总题数 128
  - 主类 7
  - 小节 30
  - 今日已测
  - 掌握率

视觉上沿用当前网站的 card-cover、主题色、轻阴影和 section-title 体系，不新建一套割裂风格。

### 总览模式

核心目标：让用户快速知道题库有什么、每类多少题、哪些题还没掌握。

布局：

- 左侧或顶部筛选栏：
  - 搜索框：题目、答案正文、关键词。
  - 主类筛选：RAG、Agent、Prompt、框架工具、微调、部署工程化、场景设计。
  - 状态筛选：未测、不会、模糊、已掌握、收藏。
  - 难度筛选：easy / medium / hard。
- 主体：
  - 分类统计卡。
  - 题目列表。
  - 每题显示：Q 编号、题目、主类、小节、关键词、掌握状态、最近一次测试结果。
- 右侧详情面板或点击进入详情：
  - 展开完整答案。
  - 按 block 展示：核心概念、原理机制、实践要点、常见追问、面试话术。
  - 支持「加入收藏」「标记掌握」「开始自测此题」。

移动端：

- 筛选栏折叠成顶部按钮。
- 题目详情使用整页抽屉或路由 hash，不做左右分栏。

### 随机自测模式

入口按钮：「随机抽题」

抽题设置：

- 抽题范围：全部 / 指定主类 / 收藏 / 错题 / 未测。
- 抽题数量：1、5、10、20。
- 模式：
  - 练习模式：不限时，答完立刻看参考答案。
  - 快问快答：每题 90 秒。
  - 弱项复盘：优先抽不会、模糊、最近答错的题。

答题流程：

1. 显示题目、分类、小节，不显示答案。
2. 用户在文本框输入自己的回答。
3. 用户点击「提交自评」。
4. 页面显示参考答案和评估面板。
5. 用户确认掌握状态，进入下一题。

随机策略：

- 同一轮内不重复抽题。
- 轮次结束前保存抽过的题目 id。
- 如果范围内题目不足，提示「本轮题库已抽完」并允许重置。
- 优先级建议：错题 > 模糊 > 未测 > 已掌握，避免纯随机一直抽到会的题。

## 评估方案

静态 GitHub Pages 不适合直接调用大模型 API，因为前端不能暴露密钥。MVP 采用「自评 + 关键词覆盖 + 结构 checklist」的组合，不依赖后端。

### MVP 评估

提交回答后显示四个维度：

1. 核心覆盖度
   - 从 `keywords` 中匹配用户回答。
   - 展示命中的关键词和遗漏关键词。
   - 分数：命中数 / 关键词总数。

2. 结构完整度
   - 根据标准 block 生成 checklist：
     - 是否讲了核心概念。
     - 是否讲了原理机制。
     - 是否讲了实践要点。
     - 是否能回答常见追问。
     - 是否能组织成面试话术。
   - 用户自己勾选，避免伪自动评分。

3. 表达成熟度
   - 字数阈值：太短提示「可能只答了定义」。
   - 是否包含项目经验词：项目、生产、指标、Recall、延迟、成本、监控、优化等。
   - 是否包含风险边界词：不足、限制、权衡、兜底、拒答、评估等。

4. 自评结果
   - 不会：0 分，进入错题。
   - 模糊：60 分，加入弱项。
   - 基本会：80 分。
   - 熟练：100 分。

最终展示一个温和的建议，而不是装作机器能准确判分：

- 「这题你覆盖了 4/7 个关键词，结构上缺少实践要点，建议再补一个项目例子。」
- 「你已经能讲清概念，但追问部分还薄，下一轮会优先再抽一次。」

### 可选 LLM 评估

后续如果有后端或本地模型，可以加 LLM 评估，但必须做成可选增强。

建议接口：

```json
{
  "questionId": "q001",
  "question": "...",
  "referenceBlocks": {...},
  "userAnswer": "...",
  "rubric": ["核心概念", "原理机制", "实践要点", "追问处理", "表达清晰度"]
}
```

输出必须是 JSON：

```json
{
  "score": 78,
  "level": "基本会",
  "missing": ["没有说明RAG不能提升推理能力", "缺少检索失败的拒答机制"],
  "strengths": ["解释了知识截止和幻觉问题"],
  "nextTip": "补一个你自己项目里的Recall@K优化例子"
}
```

注意：LLM 评估只做学习反馈，不作为真实能力判定。

## 趣味玩法

### 每日 5 题

每天自动生成一组题，按弱项优先抽。完成后显示今日完成度和连续天数。

### Boss 题

场景设计题作为 Boss 题展示，需要先答完若干基础题再解锁。比如完成 5 道 RAG 基础题后，解锁「设计一个企业知识库问答系统」。

### 错题回炉

答错或标记模糊的题会进入回炉池。下一次随机抽题时权重提高。

### 面试话术挑战

只展示题目，用户需要用 90 秒写出可直接说给面试官听的版本。提交后重点对比「面试话术」block。

## 验收标准

- 题库导入完整：128/128。
- 总览页面可以按主类、状态、关键词搜索。
- 每道题能看到完整答案，不丢代码块和表格。
- 随机抽题同一轮不重复。
- 自测记录刷新页面后仍保留。
- 无外部 API 依赖。
- 页面在手机宽度下无文字溢出、无控件重叠。
- 与当前网站视觉一致，不出现另一套设计语言。
