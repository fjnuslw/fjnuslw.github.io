A. LLM 基础（必考）

1
大模型是怎么训练出来的？请简述训练流程。
高频基础
解决思路

分三个阶段回答：预训练 → SFT 指令微调 → RLHF 对齐。这是标准答案框架。

具体答案

阶段1：预训练（Pre-training）

  • 用海量无标注文本（万亿 token 级）做 next token prediction
  • 目标函数：交叉熵损失，预测下一个 token
  • 这一步让模型学会"语言的规律"，但还不会"听指令"

阶段2：SFT（Supervised Fine-Tuning，监督微调）

  • 用"指令-回答"对（人工标注，几万到几十万条）微调
  • 让模型学会"听懂人的指令并给出符合格式的回答"
  • 这一步模型从"续写机器"变成"对话助手"

阶段3：RLHF（Reinforcement Learning from Human Feedback）

  • 训练一个奖励模型（RM），对模型输出打分
  • 用 PPO 等强化学习算法，让模型输出最大化奖励
  • 对齐人类偏好：有用、无害、诚实

现在也有用 DPO（Direct Preference Optimization） 替代 RLHF 的趋势，省去了显式的奖励模型，直接用偏好数据优化。

面试加分：提到 DPO、提到预训练数据规模、提到 RLHF 的"三阶段"（SFT+RM+PPO），会显得你对训练流程理解深入。
2
Transformer 架构的核心是什么？Encoder 和 Decoder 的区别？
高频基础
解决思路

从三个层面回答：核心机制（Self-Attention）→ 三大架构（Encoder-only/Decoder-only/Encoder-Decoder）→ 各自代表模型。

具体答案

核心机制：Self-Attention（自注意力）

  • 每个 token 和序列中所有 token 计算注意力权重
  • 公式：Attention(Q,K,V) = softmax(QKT/√dk)V
  • 解决长距离依赖问题，且可并行计算

Encoder vs Decoder 区别：

  • Encoder：双向注意力，能看到整个序列，适合理解任务（分类、抽取）。代表：BERT
  • Decoder：单向（masked）注意力，只能看到当前位置及之前，适合生成任务。代表：GPT 系列
  • Encoder-Decoder：编码器理解输入，解码器生成输出，适合 Seq2Seq（翻译）。代表：T5、BART

当前主流大模型（GPT-4、Llama、GLM 等）几乎都是 Decoder-only 架构，因为生成能力强且易于扩展。

避坑：不要只说"Transformer 是注意力机制"，要能说出 Q/K/V 三个矩阵的作用和 softmax 归一化的目的。面试官会追问"为什么要除以 √dk"（答：防止点积过大导致 softmax 梯度消失）。
3
Function Call 是怎么实现的？和 MCP 有什么区别？
高频应用
解决思路

先讲 Function Call 的实现原理，再对比 MCP（Model Context Protocol）的新特性。

具体答案

Function Call 实现流程：

  1. 定义工具：用 JSON Schema 描述工具名称、功能、参数
  2. 模型决策：把工具描述 + 用户 query 发给 LLM，LLM 判断是否需要调用工具
  3. 返回调用：如果需要，LLM 返回 {"name": "search", "arguments": {...}}
  4. 程序执行：业务代码执行真正的工具调用，拿到结果
  5. 结果回传：把工具结果作为新 message 发回 LLM，LLM 基于结果生成最终回答

MCP（Model Context Protocol）是什么？

MCP 是 Anthropic 2024 年底提出的开放协议，本质是 Function Call 的标准化升级版：

  • Function Call：每个模型厂商格式不同（OpenAI/Claude/Gemini 各搞一套）
  • MCP：统一协议，一次开发对接所有模型；支持 Resources（资源）、Prompts（提示模板）、Tools（工具）三类能力
  • MCP 是 C/S 架构：MCP Server 暴露能力，MCP Client（如 Claude Desktop）调用
# Function Call 标准工具定义
tools = [{
    "type": "function",
    "function": {
        "name": "search_web",
        "description": "搜索互联网信息",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "搜索关键词"}
            },
            "required": ["query"]
        }
    }
}]
加分：提到 MCP 是"Function Call 的 USB-C 接口"这个类比，说明它统一了协议标准。能说出 MCP 三类能力（Resources/Prompts/Tools）会更出彩。

Embedding 是什么？你们项目用的哪个模型？为什么选它？
中频RAG
解决思路

先解释概念，再结合项目实际回答选型理由。即使没做过项目，也要准备好一个"为什么选这个模型"的故事。

具体答案

Embedding 是什么：把文本映射成稠密向量（如 768/1536 维），让语义相近的文本在向量空间中距离也近。是 RAG 检索的基础。

常见 Embedding 模型选型：

  • OpenAI text-embedding-3-small/large：效果好、成本低，1536/3072 维
  • BGE 系列（智源）：中文效果好，开源可本地部署，MTEB 榜单常年前列
  • M3E：中文场景常用，开源
  • Qwen Embedding：阿里通义系列，多语言支持好
  • Cohere embed-v3：多语言，英文场景强

选型理由（参考话术）：

  • "我们项目主要处理中文文档，选了 BGE-large-zh，因为 MTEB 中文榜单第一，且可本地部署，数据不出域"
  • "对成本敏感的场景用 OpenAI text-embedding-3-small，单价便宜且效果够用"
  • "需要多语言时选 Cohere embed-v3 或 Qwen Embedding"
准备建议：至少准备好一个 Embedding 模型的选型故事，能说出"为什么选、效果如何、有没有对比过其他方案"。

B. Prompt 工程

5
ReAct 是什么？怎么实现的？
高频Agent
解决思路

ReAct = Reasoning + Acting，是 Agent 最经典的范式。讲清楚"思考-行动-观察"循环即可。

具体答案

ReAct 核心思想：让 LLM 交替进行"推理（Thought）"和"行动（Action）"，并通过"观察（Observation）"获取反馈，直到完成任务。

实现流程（循环）：

Thought: 我需要先搜索相关信息
Action: search_web("2026大模型趋势")
Observation: [搜索结果...]
Thought: 根据结果，我需要再查一下数据
Action: search_web("大模型市场规模 2026")
Observation: [搜索结果...]
Thought: 信息已足够，可以生成答案了
Action: finish("2026年大模型行业趋势报告...")

Prompt 模板示例：

你是一个能使用工具的助手。请按以下格式回答：

Question: {用户问题}
Thought: 你对下一步该怎么做的思考
Action: 工具名称
Action Input: 工具参数
Observation: 工具返回结果
...(重复 Thought/Action/Observation)
Thought: 我现在知道答案了
Final Answer: 最终回答
对比 CoT：CoT 只有"思考链"，没有"行动"和"外部反馈"，适合纯推理任务；ReAct 能调用工具获取实时信息，适合需要外部数据的任务。面试时主动对比会加分。
6
CoT 是什么？为什么效果好？有什么缺点？
高频Prompt
具体答案

CoT（Chain of Thought，思维链）：在 Prompt 中引导 LLM "一步一步思考"，输出中间推理过程再给答案。

为什么效果好：

  • 分解复杂问题：把多步推理拆成单步，降低每步难度
  • 增加生成 token 数：给模型更多"计算预算"，中间 token 相当于"草稿纸"
  • 对齐训练数据：很多训练数据本身包含推理过程

经典触发方式：

  • Zeor-Shot：让我们一步一步思考
  • Few-Shot：给几个带推理过程的示例

缺点：

  • 延迟增加：输出 token 变多，响应变慢
  • 成本增加：token 消耗翻倍
  • 错误传播：中间步骤错了，后续全错
  • 不一定有效：对简单任务反而可能降低准确率
进阶：提到 ToT（Tree of Thoughts，思维树）和 GoT（Graph of Thoughts），它们是 CoT 的升级版，支持分支探索和回溯。
7
温度值（Temperature）、top-p、top-k 分别是什么？不同场景怎么设？
高频基础
具体答案

三者都是控制 LLM 生成"随机性/多样性"的参数，本质都在调整采样概率分布。

Temperature（温度）

  • 作用：对 logits 除以 T 后再 softmax。T 越小分布越尖锐（更确定），T 越大越平坦（更随机）
  • 范围：通常 0-2。0 = 贪心解码（几乎确定），1 = 原始分布，>1 增加随机性

top-p（核采样 Nucleus Sampling）

  • 作用：从概率高到低累加，累计概率达到 p 时停止，只从这个最小集合中采样
  • p=0.9：只从累计概率占 90% 的 top token 中采样，过滤掉长尾低概率 token

top-k

  • 作用：只从概率最高的 k 个 token 中采样
  • k=40：只考虑 top 40 个候选 token

不同场景推荐设置：

场景 Temperature top-p 说明
代码生成01要确定性
RAG 问答0-0.31忠实于检索内容
客服对话0.5-0.70.9自然但可控
创意写作0.8-1.00.95鼓励多样性
Function Call01工具选择要确定
实战经验：OpenAI 官方建议"只调 temperature 或 top-p 中的一个，不要同时调"，否则效果难控。

C. 框架与工具

8
介绍一下 LangChain。为什么手搓 Agent 而不用框架？
中频框架
具体答案

LangChain 核心模块：

  • Models：统一封装各 LLM 接口（OpenAI/Claude/本地模型）
  • Prompts：Prompt 模板管理
  • Chains：把多个步骤串联（如先检索再生成）
  • Agents：基于 LLM 决策调用工具
  • Memory：对话记忆管理
  • Retrievers：RAG 检索器

为什么手搓而不用框架？（高频追问）

  • 可控性：框架封装太厚，出 bug 难定位；手搓每一步都自己掌控
  • 灵活性：业务场景特殊时，框架限制多；手搓可自由定制
  • 性能：框架有额外开销（抽象层、序列化）；手搓可针对性优化
  • 学习：手搓能真正理解 Agent 原理，不被框架"绑架"
  • 依赖：LangChain 版本迭代快，API 不稳定；手搓无依赖风险
推荐回答策略："小项目/原型用 LangChain 快速验证，生产环境/复杂场景手搓以保证可控性和性能"。这样既显得懂框架，又显得有工程判断力。
9
MCP 是什么？A2A 了解吗？
中频前沿
具体答案

MCP（Model Context Protocol）：见第 3 题，是 Function Call 的标准化协议升级。

A2A（Agent to Agent）：Agent 之间相互通信协作的协议/模式。

  • Google 提出的 A2A 协议：让不同厂商的 Agent 能互相发现、协商、协作
  • 典型场景：一个 Agent 负责查数据，另一个负责生成报告，A2A 让它们自动对接
  • 对比 Multi-Agent 框架（AutoGen/CrewAI）：那些是"一个系统内的多 Agent"，A2A 是"跨系统的 Agent 互联"
加分：提到 MCP 是"工具调用层的标准"，A2A 是"Agent 互联层的标准"，两者层次不同，会显得理解很清晰。

D. RAG 核心（一面必问）

10
介绍一下 RAG。最难的地方在哪？
必考RAG
解决思路

先讲 RAG 是什么（2 分钟），再重点讲难点。难点答得好基本这一题就过了。

具体答案

RAG（Retrieval Augmented Generation，检索增强生成）：先从知识库检索相关文档，再把检索结果作为 context 喂给 LLM 生成答案。

标准流程：文档预处理 → 切分 → Embedding → 存入向量库 → query 向量化 → 相似度检索 → 拼 Prompt → LLM 生成

三大难点（按重要性）：

难点1：文档切割（最高频答案）

  • 切太粗：检索到太多无关内容，浪费 context、稀释信号
  • 切太细：语义不完整，模型无法理解
  • 切割位置不当：把一句话/一个段落从中间切断，语义断裂
  • 解决方案：语义切分（按段落/标题）、滑动窗口+overlap、递归切分、基于小模型的语义边界检测

难点2：召回质量

  • 向量检索召回不准（query 和文档语义鸿沟）
  • 解决方案：多路召回（向量+BM25+关键词）、query 改写/扩展、HyDE（假设性文档嵌入）

难点3：生成忠实度（防幻觉）

  • LLM 可能"无视"检索内容自己编
  • 解决方案：强约束 Prompt（"只基于以下内容回答"）、引用标注、置信度检测、RAGAS 评测
面试话术："我觉得 RAG 最难的是文档切割，因为它直接影响后续所有环节的效果。其次是召回质量和防幻觉。我在项目里用的是 XXX 切分策略..."。结合自己项目讲最加分。
11
文档切割策略有哪些？怎么避免语义被切断？
高频RAG
具体答案

常见切割策略（从简单到复杂）：

  1. 固定长度切分：按字符数（如 500 字）切。缺点：可能切断句子/段落
  2. 按分隔符切分：按 \n\n、.、。 等切。优点：保留句子完整
  3. 滑动窗口 + Overlap：固定长度但相邻块有重叠（如 500 字块，100 字 overlap）。优点：边界信息不丢
  4. 递归切分：先按大分隔符（段落）切，太长再按小分隔符（句号）切。LangChain 的 RecursiveCharacterTextSplitter
  5. 语义切分（Semantic Chunking）：用 Embedding 计算相邻句子相似度，相似度骤降处切分。效果最好但成本高
  6. 基于文档结构：Markdown 按 # 标题层级切，PDF 按页/章节切，HTML 按标签切
  7. 小模型辅助：用小模型识别语义边界（如句号、段落结束），比纯规则更准

避免语义断裂的实战技巧：

  • 切分后保留 chunk 的上下文元数据（所属章节、前后文摘要）
  • 检索时不仅返回 chunk，还返回上下文（Parent Document Retrieval）
  • 对长文档做摘要后再切分（Summary Index）
避坑：不要只说"按字数切"，面试官会觉得太浅。至少要讲出 3 种以上策略，并说出各自的优缺点和适用场景。
12
怎么解决大模型的幻觉问题？
必考RAG
具体答案

幻觉（Hallucination）：LLM 生成看似合理但实际错误/虚构的内容。分两类：事实性幻觉（说错事实）、忠实性幻觉（不忠于源文档）。

缓解策略（多层防御）：

1. 检索层（RAG）

  • 提供高质量、权威的外部知识，让模型"有据可依"
  • 多路召回提升召回率，确保相关信息被检索到

2. Prompt 层

  • 强约束：只根据以下参考资料回答，如果资料中没有答案，请说"我不知道"
  • 要求引用：每个论断都要标注来源 [1][2]
  • CoT：让模型先分析资料再回答

3. 模型层

  • 用 RLHF 对齐过的模型（如 GPT-4、Claude）
  • 降低 temperature（0-0.3）减少随机生成
  • 微调（SFT）让模型学会"说不知道"

4. 后处理层

  • 用另一个 LLM 做 fact-check（自我审查）
  • 引用验证：检查生成内容是否真的能在源文档找到
  • 置信度评分：低置信度时拒答或转人工

5. 评测层

  • 用 RAGAS 框架评测 faithfulness（忠实度）指标
  • TruLens 等工具做线上幻觉监控
关键认知：幻觉无法消除只能缓解。面试时强调"多层防御"和"评测闭环"会显得很专业。
13
微调和 RAG 的优劣势？怎么选？
高频综合
具体答案
维度 RAG 微调
知识更新实时，改库即可慢，需重新训练
成本低（向量库+检索）高（GPU训练）
知识可解释强（有引用）弱（黑盒）
风格/格式弱强（可定制语气）
幻觉控制强（有据可查）弱
领域知识量海量（TB级）有限（训练数据）
推理速度慢（多一步检索）快

选择建议：

  • 选 RAG：知识频繁更新、需要引用溯源、知识量大、预算有限
  • 选微调：需要特定风格/格式、领域术语多、对延迟敏感、任务模式固定
  • 两者结合：微调让模型懂"领域语言"，RAG 提供"领域知识"。生产环境常组合使用
14
多路召回是什么？为什么要用图数据库？
中频RAG
具体答案

多路召回（Hybrid Retrieval）：同时用多种检索方式，融合结果提升召回率。

  • 向量检索：语义相似，擅长理解同义/近义
  • BM25（关键词）：词项匹配，擅长精确关键词（产品名、人名、代码）
  • 元数据过滤：按时间/来源/标签筛选

融合方法：RRF（Reciprocal Rank Fusion）最常用，公式简单效果好。

图数据库在 RAG 中的作用（GraphRAG）：

  • 向量检索只能找"相似"文档，无法处理"关联关系"问题
  • 图数据库存储实体关系（A 是 B 的老板，C 属于 D 部门）

怎么量化 RAG 的效果？检索效果和回答效果怎么评估？

具体答案

RAGAS 框架（最主流）四大指标：

  • Faithfulness（忠实度）：回答是否忠于检索内容，有无幻觉。用 LLM 判断每个论断是否可被 context 支持
  • Answer Relevancy（答案相关性）：回答是否切题。用 LLM 反向生成问题，与原 query 比对
  • Context Precision（上下文精确率）：检索到的文档中有多少是真正相关的
  • Context Recall（上下文召回率）：相关文档是否都被检索到

检索层指标：

  • Recall@K：top-K 中包含正确文档的比例
  • MRR（Mean Reciprocal Rank）：正确文档排名倒数的均值
  • nDCG：考虑排名位置的增益

生成层指标：

  • BLEU/ROUGE：与标准答案的 n-gram 重叠（局限大）
  • LLM-as-a-Judge：用 GPT-4 打分（主流）
  • 人工评估：准确率、满意度（最终标准）

业务指标：

  • 用户反馈（点赞/点踩率）
  • 任务完成率
  • 人工接管率

算法题：给定数组，每次取一个数 K，移除所有 K-1 和 K+1，如何取使所取数之和最大？小红书真题
DPMedium
解决思路

这是经典的"打家劫舍"变形。统计每个数字的总和（值×频次），然后 DP：取 nums[i] 就不能取 nums[i-1] 和 nums[i+1]。

具体答案
def deleteAndEarn(nums):
    if not nums: return 0
    max_val = max(nums)
    points = [0] * (max_val + 1)
    for num in nums:
        points[num] += num  # 每个值累加分

    # 打家劫舍 DP
    prev, curr = 0, points[0]
    for i in range(1, len(points)):
        prev, curr = curr, max(curr, prev + points[i])
    return curr

# 时间 O(max(nums))，空间 O(max(nums))

算法题：给定整数数组和整数 k，找到和为 k 的连续子数组个数。小红书真题
前缀和Medium
解决思路

前缀和 + 哈希表。如果 prefix[j] - prefix[i] = k，则 [i+1, j] 是目标和为 k 的子数组。

具体答案
def subarraySum(nums, k):
    count = 0
    prefix_sum = 0
    prefix_count = {0: 1}  # 前缀和0出现1次

    for num in nums:
        prefix_sum += num
        # 找之前有多少个前缀和等于 prefix_sum - k
        if (prefix_sum - k) in prefix_count:
            count += prefix_count[prefix_sum - k]
        prefix_count[prefix_sum] = prefix_count.get(prefix_sum, 0) + 1

    return count

# 时间 O(n)，空间 O(n)

算法题：无重复字符的最长子串长度。字节真题
滑动窗口Medium
解决思路

滑动窗口 + 哈希表记录字符最后出现位置。右指针扩展，遇到重复字符则左指针跳到重复字符下一位。

具体答案
def lengthOfLongestSubstring(s):
    char_index = {}
    left = 0
    max_len = 0

    for right, char in enumerate(s):
        if char in char_index and char_index[char] >= left:
            left = char_index[char] + 1
        char_index[char] = right
        max_len = max(max_len, right - left + 1)

    return max_len

# 时间 O(n)，空间 O(min(n, 字符集大小))

算法题：最长递增子序列（LIS）。字节真题

解决思路

两种解法：O(n²) DP 或 O(n log n) 二分。面试先说 DP，再主动提二分优化。

具体答案
# 方法1：DP O(n²)
def lengthOfLIS(nums):
    if not nums: return 0
    dp = [1] * len(nums)
    for i in range(1, len(nums)):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp)

# 方法2：二分 + 贪心 O(n log n)
import bisect
def lengthOfLIS(nums):
    tails = []
    for num in nums:
        idx = bisect.bisect_left(tails, num)
        if idx == len(tails):
            tails.append(num)
        else:
            tails[idx] = num
    return len(tails)
# tails[i] 表示长度为 i+1 的递增子序列的最小末尾

具体答案
def minDistance(word1, word2):
    m, n = len(word1), len(word2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    for i in range(m + 1): dp[i][0] = i
    for j in range(n + 1): dp[0][j] = j

    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if word1[i-1] == word2[j-1]:
                dp[i][j] = dp[i-1][j-1]
            else:
                dp[i][j] = 1 + min(dp[i-1][j],    # 删除
                                   dp[i][j-1],    # 插入
                                   dp[i-1][j-1])  # 替换
    return dp[m][n]

# 空间优化：用滚动数组 O(min(m,n))

算法题：三数之和。字节真题
高频双指针
具体答案
def threeSum(nums):
    nums.sort()
    res = []
    n = len(nums)

    for i in range(n - 2):
        if i > 0 and nums[i] == nums[i-1]:
            continue  # 去重
        left, right = i + 1, n - 1
        while left < right:
            total = nums[i] + nums[left] + nums[right]
            if total == 0:
                res.append([nums[i], nums[left], nums[right]])
                while left < right and nums[left] == nums[left+1]: left += 1
                while left < right and nums[right] == nums[right-1]: right -= 1
                left += 1
                right -= 1
            elif total < 0:
                left += 1
            else:
                right -= 1
    return res

# 时间 O(n²)，空间 O(1)

# 方法2：快速选择 O(n) 平均
import random
def findKthLargest(nums, k):
    def partition(left, right):
        pivot_idx = random.randint(left, right)
        nums[pivot_idx], nums[right] = nums[right], nums[pivot_idx]
        pivot = nums[right]
        i = left
        for j in range(left, right):
            if nums[j] > pivot:  # 降序，找第k大
                nums[i], nums[j] = nums[j], nums[i]
                i += 1
        nums[i], nums[right] = nums[right], nums[i]
        return i

    left, right = 0, len(nums) - 1
    while True:
        pos = partition(left, right)
        if pos == k - 1:
            return nums[pos]
        elif pos > k - 1:
            right = pos - 1
        else:
            left = pos + 1

Agent 的反思机制是什么？为什么要用？怎么实现？
中频Agent
具体答案

反思（Reflection）：让 Agent 回顾自己的执行过程，总结经验教训，用于改进后续决策。

为什么要用：

  • LLM 一次决策可能出错，反思能让它"自我纠错"
  • 把"经验"存入记忆，下次遇到类似任务能复用
  • 提升复杂任务的完成率（Reflexion 论文证明提升明显）

  2. 过程中反思：每执行N步，让 LLM 检查"是否偏离目标/有没有更好方案"
  3. 失败反思：任务失败时，让 LLM 分析原因并重试（Reflexion 模式）

Python 的内存管理和 GC 机制？
八股
具体答案

内存管理：Python 用私有堆空间存储对象，解释器负责管理。

  • 对象池：小整数（-5~256）、短字符串 intern 机制，复用对象
  • 内存分配器：pymalloc，针对小对象（≤512字节）用内存池，大对象直接 malloc
  • 引用计数：每个对象有 ob_refcnt，引用+1/-1，归0立即释放

GC 机制（三世代回收）：

  • 引用计数（主要）：实时释放无引用对象。缺点：无法处理循环引用
  • 标记-清除（辅助）：解决循环引用。从根对象出发标记可达对象，清除不可达
  • 分代回收：对象分三代，新对象在第0代，存活越久越老，老对象检查频率低

GC 触发条件：第0代分配数 - 释放数 > 阈值（默认700）。

zset（有序集合）实现：ziplist + skiplist（Redis 7.0后用 listpack）

  • 元素少时（128个且每个≤64字节）：用 ziplist/listpack，紧凑存储省内存

  • 范围查询友好：skiplist 天然有序，范围查询 O(logN)
  • 并发友好：skiplist 局部修改，红黑树可能全局 rebalance
  • 内存：skiplist 多指针有一定开销，但可接受

MySQL 的 MVCC 是怎么实现的？
八股
具体答案

MVCC（Multi-Version Concurrency Control）：多版本并发控制，让读写不互相阻塞。

核心组件：

  1. 隐藏字段：每行有 DB_TRX_ID（最后修改的事务ID）、DB_ROLL_PTR（回滚指针指向 undo log）、DB_ROW_ID
  2. undo log：存历史版本，通过回滚指针串成版本链
  3. ReadView：读事务创建的快照，记录当时活跃事务列表

    4. 可见性判断（RC vs RR）：

    • RC（读已提交）：每次 SELECT 都创建新 ReadView → 能读到最新已提交数据
    • RR（可重复读）：事务开始时创建一次 ReadView → 整个事务读同一快照

    版本链遍历规则：从最新版本开始，判断 DB_TRX_ID 是否对当前 ReadView 可见：

    • 已提交且早于 ReadView：可见
    • 在 ReadView 活跃事务中：不可见，顺指针找历史版本
    • 晚于 ReadView：不可见，找历史版本

手写 Multi-Head Attention字节真题
大模型特色PyTorch
import torch
import torch.nn as nn
import torch.nn.functional as F

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.d_model = d_model
        self.n_heads = n_heads
        self.d_k = d_model // n_heads

        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)

    def forward(self, x, mask=None):
        batch_size, seq_len, _ = x.size()

        # 线性变换 + 分头
        Q = self.W_q(x).view(batch_size, seq_len, self.n_heads, self.d_k).transpose(1, 2)
        K = self.W_k(x).view(batch_size, seq_len, self.n_heads, self.d_k).transpose(1, 2)
        V = self.W_v(x).view(batch_size, seq_len, self.n_heads, self.d_k).transpose(1, 2)

        # 注意力分数
        scores = torch.matmul(Q, K.transpose(-2, -1)) / (self.d_k ** 0.5)

        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))

        # softmax
        attn = F.softmax(scores, dim=-1)

        # 加权求和
        context = torch.matmul(attn, V)

        # 合并头
        context = context.transpose(1, 2).contiguous().view(batch_size, seq_len, self.d_model)

        return self.W_o(context)
关键点：①分头维度变换（view+transpose）②缩放 ③mask 处理 ④合并头。面试官会追问 RoPE 位置编码怎么加。

手写 KV Cache 逻辑字节真题
大模型特色推理优化
解决思路

自回归生成时，每生成一个新 token，之前的 K/V 不变，缓存起来避免重复计算。

class AttentionWithKVCache:
    def __init__(self, d_model, n_heads):
        self.n_heads = n_heads
        self.d_k = d_model // n_heads
        self.cache_k = None  # [batch, heads, seq, d_k]
        self.cache_v = None

    def forward(self, x_new, use_cache=True):
        # x_new: [batch, 1, d_model] 每次只输入新token
        Q = self.W_q(x_new)  # 只算新token的Q
        K_new = self.W_k(x_new)
        V_new = self.W_v(x_new)

        if use_cache and self.cache_k is not None:
            # 拼接历史KV
            K = torch.cat([self.cache_k, K_new], dim=2)
            V = torch.cat([self.cache_v, V_new], dim=2)
        else:
            K, V = K_new, V_new

        # 更新缓存
        self.cache_k = K
        self.cache_v = V

        # 注意力计算
        scores = Q @ K.transpose(-2, -1) / (self.d_k ** 0.5)
        attn = F.softmax(scores, dim=-1)
        return attn @ V

核心思想：生成第 t 个 token 时，只需要计算新 token 的 Q，K/V 用缓存的+新的。把 O(n²) 的重复计算降到 O(n)。

import torch
import torch.nn.functional as F

def sft_loss(logits, labels, ignore_index=-100):
    # logits: [batch, seq_len, vocab_size]
    # labels: [batch, seq_len]

    # 关键：shift一位，因为要预测下一个token
    # logits[:, :-1] 预测 labels[:, 1:]
    shift_logits = logits[..., :-1, :].contiguous()
    shift_labels = labels[..., 1:].contiguous()

    # 展平计算交叉熵
    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),
        shift_labels.view(-1),
        ignore_index=ignore_index  # padding位置不计loss
    )
    return loss

  2. 不会就说不会：别编一个自己都听不懂的答案，真诚 > 装懂
  3. 简历别吹过头：可以用大模型润色，但要自己核对，面试一追问就露馅
  4. 多看技术文章：二面面试官很看重技术广度，多关注前沿
  5. 表达流畅清晰：断断续续会被认为思路不清
  6. 项目效果必须量化：不管技术多炫，最终看落地效果。准备好"准确率从X%到Y%"这种数据
高频踩坑点
  • 八股不准备：虽然少了但绝对不能不准备，有人多次挂在八股
  • 只会调 API：面试官最反感"调包侠"，要懂原理（微调原理是大加分项）
  • 项目讲不深：只说"用了LangChain做了RAG"，一问细节就卡壳
  • 没有量化：说"效果不错"但不给数据，面试官不信
  • 算法题写对就放松：字节即使写对也可能挂，要看沟通和思路
  • 期望薪资报太高：对方说"再等等"基本就是凉了
针对双非本硕+论文+国赛的特别建议
  • 扬长避短：简历突出论文+国赛，弱化学校。项目经历放前面
  • 项目要能讲深：双非过简历关靠项目，面试靠项目讲透。准备2-3个有深度的项目
  • 算法题必须稳：大厂一面必考算法，Bug Free 是底线。重点刷 DP 和滑动窗口
  • 论文是加分项：主动提，但别炫技。说清楚"解决了什么问题、用了什么方法、效果如何"
  • 国赛要能讲清思路：面试官会问竞赛题的解法，提前准备好讲稿
  • 心态：双非面大厂简历可能被卡，多投多试，别因几次简历挂就放弃
