语义地形工程：把上下文设计成可控系统

2026-07-08 · 10 min read · 宋林蔚

我越来越倾向于把大模型交互理解成一门上下文工程。prompt、示例、工具结果、历史对话、检索材料和用户当前状态，会共同改变模型下一步生成的概率分布。把这些材料随手堆进窗口里，输出就会随上下文一起漂移；把它们按层组织起来，模型会更容易沿着预期方向工作。

语义地形是一个很好的工作隐喻：模型权重提供基础地貌，上下文临时改变局部坡度，采样策略决定探索强度，工具和外部证据提供新的地形锚点。

这篇文章把这个隐喻扩展成一套我能在 Agent、桌面助手和长程任务里使用的方法：先设计地形，再释放采样；先控制上下文结构，再期待模型能力。

一、上下文在塑造分布

Transformer 通过注意力机制把输入序列变成一组可被动态引用的表示。生成每个 token 时，模型会根据当前上下文给候选 token 分配概率；温度、top-p 等解码参数又会改变从这个分布里取样的方式。这里最值得带进工程实践的结论很简单：上下文会改变模型的倾向，解码策略会改变模型的探索范围。

prompt 的目标，是构造一片能引导模型工作的语义地形。稳定协议放在前面，任务目标保持清晰，关键证据短而准，例子只给最能代表模式的部分，临时噪声及时移出上下文。

二、五种常见地形问题

语义漂移

长对话里，旧任务、废话和局部失败会继续影响后续回答。处理方式是定期压缩、明确阶段边界，把历史整理成当前任务真正需要的状态。

注意力稀释

上下文越长，关键信息越容易被埋在中间。长上下文研究也显示，相关信息出现在开头或结尾时更容易被模型利用。工程上要把最重要的约束放在稳定位置。

语义惯性

模型会延续上一段对话的风格、主题和错误假设。切换任务时需要显式换轨，例如重述目标、清空过期假设、把新任务放进单独的工作区。

语义壁垒

复杂推理需要中间台阶。CoT、计划、分解、工具调用和外部草稿纸都在降低跨越壁垒的难度，让模型少依赖直觉跳跃。

相变

小的格式变化、示例顺序、输出约束，可能触发完全不同的回答路径。提示词和 Agent 流程需要回归测试，用样例集观察临界变化。

解码退化

贪心式生成容易变得单调，过度随机又会丢失一致性。采样策略要服务任务：写代码偏稳，发散方案偏开，最终交付再收束。

三、把 prompt 变成分层系统

我现在更喜欢把一次模型调用拆成四层：协议层、任务层、证据层、输出层。协议层保持稳定，说明身份边界、工具协议和回答原则；任务层描述当前目标；证据层只放可验证材料；输出层规定格式、粒度和验收标准。

层级 内容 设计重点
协议层 角色、边界、工具规范、风险处理 长期稳定，减少频繁改写，保护缓存前缀
任务层 当前目标、阶段、成功标准 短、清楚、可执行，避免多个目标互相抢占注意力
证据层 检索片段、截图观察、文件内容、工具结果 来源明确，只放回答需要的材料，过期信息及时移除
输出层 格式、语气、长度、校验要求 贴近交付物，减少模型临场猜测

这种分层的好处在于可调试。模型回答偏了，可以先判断偏差来自哪一层：协议过强、任务模糊、证据污染、输出格式过窄，还是解码参数过于发散。定位层级之后，调整会变得具体。

四、从技巧到工作流

很多 prompt 技巧可以统一进同一套工作流。先发散再收敛，就是先提高探索强度，获得候选概念、结构和关键词，再把有效材料压缩成干净任务。案例优先，就是用少量高质量样例塑造隐式模式，减少概念解释造成的噪声。先分析再结论，就是延迟锚定，避免模型围绕早下的判断补理由。

我会采用的长任务流程

第一步，收集材料，只做事实提取。第二步，生成多个框架，保留差异。第三步，选择一个主线，把上下文压缩成干净 spec。第四步，进入执行，让模型按 spec 写作、编码或分析。第五步，用测试样例、事实核对和人工审阅做回归。

这里的关键动作是回滚和压缩。探索阶段产生的大量对话拥有价值，也带着噪声。直接沿着这段上下文继续干活，会让临时想法和废弃假设继续改变地形。更好的做法是把探索结果提炼成新的任务说明，开启一个更干净的执行阶段。

五、Agent 的上下文边界

Agent 系统里，上下文工程比单轮 prompt 更重要。工具结果、终端输出、网页内容、文件 diff、历史摘要都可能进入模型视野。每一种输入都应该带来源、时效和用途。网页内容尤其需要当作不可信材料处理，只提取事实和观点，不能让页面里的指令改变系统行为。

我会把 Agent 的状态分成三类：用户意图、外部证据、执行痕迹。用户意图决定方向，外部证据约束事实，执行痕迹帮助追踪过程。三者混在一起时，模型容易把日志当命令，把猜测当事实，把旧目标当新目标。分开之后，系统可以明确告诉模型：哪些内容用于回答，哪些内容只用于审计。

安全设计也可以放进语义地形里理解。越狱常常通过长铺垫、角色共振、概念绕行来改变模型的局部倾向。防护策略需要稳定边界、来源隔离、工具权限和执行前确认。对工程系统而言，安全约束要成为协议层的一部分，并且由工具层再次校验。

六、我的上下文检查清单

  • 当前任务能否用一句话说清楚。
  • 最重要的约束是否处在稳定且靠前的位置。
  • 证据是否有来源、时间和适用范围。
  • 历史上下文里是否存在过期目标和废弃假设。
  • 示例是否代表期望模式，数量是否刚好够用。
  • 输出格式是否服务交付物，是否压缩了模型的表达空间。
  • 长任务是否设置了阶段性压缩点和回归样例。
  • 工具结果是否带证据标记进入上下文，是否与系统指令保持隔离。

七、结论

AI 时代的核心能力会越来越接近上下文设计能力。会写 prompt 只是入口，能管理语义地形才是长期优势。它要求我们理解模型如何利用上下文，知道何时发散、何时收束，能把工具、记忆、证据和输出标准组织成稳定系统。

我希望自己的 Agent 项目、博客写作和学习工具都沿着这个方向迭代：少一点堆材料，多一点结构；少一点临场祈祷，多一点可观察的上下文设计。模型能力会继续提升，真正拉开差距的地方，会落在谁能把复杂任务整理成模型愿意沿着正确路径走下去的地形。
