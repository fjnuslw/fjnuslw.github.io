Agent VLM 架构 复盘

小 VLM 的 function calling 有多不稳定：一次 probe→stream 架构重构

2026-07-05 · 15 min read · 宋林蔚

上一篇文章写了桌面 AI 伙伴的上下文管理踩坑，这篇是第二天继续调试的复盘。 如果说上一篇的主题是"架构怎么分"，这篇的主题就是"分了之后，每一层都会出什么你意想不到的问题"。

项目有两条链路：观察线（静默截图 → VLM 分析 → 存 SQLite）和对话线（用户提问 → 工具检索 → 生成回答）。 两条链路都坏了，但坏的方式完全不同。观察线的 VLM 成功率只有 54%，对话线则"能跑但不干活"—— 模型知道该调工具，却不调，或者调了之后不回答。

这篇文章记录我是怎么一步步定位这两个问题的根因的，以及为什么最终的修复方式不是"加 fallback" 而是"重构职责边界"。

一、观察线：46% 失败率背后的退化运行

先看观察线。SQLite 里有 26 次成功、22 次失败，失败率 46%。 我最初以为是 max_tokens 不够，于是从 8192 调到 16384，失败率几乎没变。

真正的发现来自对比成功和失败的输出长度：

状态输出 tokens耗时输出内容
成功1000–17548–11s结构化 JSON，字段完整
失败16384（打满）~110s退化重复，同一句话循环

失败模式不是"token 不够"，而是模型退化重复——它在一个短语上无限循环， 直到打满 max_tokens 才停。增大 max_tokens 不仅没用，反而让单次失败从 30 秒拖到 110 秒。

误区警示：当模型输出退化重复时，增大 max_tokens 是最坏的选择。 它不解决问题，只是让失败更慢。正确的方向是减少 prompt 的结构复杂度，让模型在稳定输出范围内完成任务。

根因在 prompt 契约。观察 prompt 要求 VLM 输出 10+ 个嵌套字段 （key_points[6-12]、regions[4-8] 每个 region 里还有 visible_text[2-8] 和 ui_elements[2-8]）。 MiniCPM-V 是一个小模型，它的稳定输出能力远低于这个字段密度。 当 prompt 要求的复杂度超过模型的稳定输出范围，它就开始退化。

修复方式是精简 prompt 契约，把字段下限降到模型实际能稳定输出的范围： key_points 6-12 → 3-6，regions 4-8 → 3-5， visible_text 8-30 → 5-15。同时 max_tokens 回归 8192。 修复后成功率从 54% 提升到 80%。

核心认知：小模型的 prompt 契约不能照搬大模型的字段密度。 prompt 要求的字段数量必须匹配模型的稳定输出能力，否则不是"少输出几个字段"，而是"整体退化"。

二、对话线：排名器复用 VLM 导致连锁失败

对话线的问题更隐蔽。用户提问时，模型通过 memory.search(query) 工具检索窗口观察记录。 但这个工具的 ranker（排名器）复用了观察线的同一个 VLM——要求模型输出 {"ids":[...], "notes":{...}} 的 JSON 来对候选证据排序。

问题很明显：VLM 在观察线上有 46% 概率无法稳定输出 JSON，在排名器上同样不稳定。 于是形成了连锁失败：

修复方式是用确定性算法替代 LLM ranker。 候选集只有 30-40 条（窗口观察 30 条 + 最新 1 条 + 记忆数条）， 这个规模下 BM25 的区分度完全够用，而且零失败率、毫秒级延迟。 中文采用 bigram（双字滑窗）分词，不需要外部分词器依赖。

指标VLM ranker（旧）FTS5 BM25（新）
失败率46%0%
延迟12-30s<10ms
依赖同一 VLM内存 SQLite
连锁风险有（VLM 挂 = ranker 挂）无（完全解耦）

这一步修复后，memory.search 工具零失败。两条链路彻底解耦—— 观察线 VLM 挂了不影响对话线的工具检索，对话线不再依赖任何模型调用就能完成证据排名。

三、probe content 被当答案：小 VLM function calling 的隐藏陷阱

修完前两个问题后，我遇到了一个更诡异的 bug。用户问"你确认一下里面都有什么吧"， 桌宠回答："调用 memory.search 检查窗口内容。"——然后就没了。

我：你确认一下里面都有什么吧
桌宠：调用 memory.search 检查窗口内容。
我：……然后呢？

它把工具调用的意图描述当成了最终答案返回。这听起来很蠢，但定位过程很有意思。

对话线的设计是两阶段：先做一个 probe 调用（带 tools 参数），让模型决定是否调 memory.search； 如果 probe 返回了 tool_calls，执行工具后进入 stream 生成答案；如果没有 tool_calls， 原代码直接把 probe 返回的 content 当答案。

问题在于：MiniCPM-V 的 function calling 不稳定。有时候它正确返回结构化 tool_calls， 有时候它把调用意图写成自然语言放在 content 里 （比如"调用 memory.search 检查窗口内容"），而不是结构化字段。 原代码看到 tool_calls 为空，就把这句自然语言当答案返回了。

我查了 SQLite 里的交互轨迹，对比了成功和失败的 session：

Session问题probe 返回结果
4cbfc5a6"当前窗口是？"tool_calls ✓正确调工具 → 生成答案
f8a6d000"你确认一下里面都有什么吧"content="调用memory.search..." ✗意图描述当答案返回

两条路径的唯一区别就是模型有没有正确发出结构化 tool_calls。这不是 prompt 能控制的—— 同一个 prompt，模型有时发结构化字段，有时写自然语言，这是小模型 function calling 的固有不确定性。

修复方式是职责分离：probe 阶段的 content 一律不信任。 probe 的唯一职责是决策是否调工具，只有 tool_calls 是它的有效产出。 无论 probe 返回什么 content，都丢弃，最终答案由 stream 阶段生成。

设计原则：当模型的某种输出格式不稳定时，不要试图"修好"它， 而是让架构不依赖那个不稳定的输出。probe 的 content 不可信， 就不让它承担产出答案的职责。这不是 fallback，是职责边界。

四、对话惰性：15642 字符证据只换回 99 字回答

修完 probe 之后又发现新问题。有一条对话：用户问"当前 codex 有什么问题"， 模型成功调了 memory.search，拿到了 4 条高度相关的窗口观察记录， 共 15642 字符——里面有 summary、key_points、regions、visible_text、ui_elements， 证据非常丰富。但模型最终只输出了 99 个字。

15642 字符的丰富证据，只换回 99 字的泛泛回答。这不是模型能力问题，是两个隐藏根因叠加：

根因 A：stream 阶段没有 tools，probe 失败 = 永远没第二次机会

原架构中，只有 probe 阶段带 tools 参数，stream 阶段不带。 如果 probe 没发出 tool_calls（上面那个问题），stream 阶段模型手里没有任何工具， 只能凭训练数据编。即使 probe 成功调了工具，stream 也只是"带着工具结果生成文本"， 模型在 stream 中无法再发起工具调用。

修复：stream 阶段也带 tools。如果模型在流式输出中发起了 tool_calls， 代码自动执行工具、把结果注入 messages、重新 stream。最多循环 2 轮。 这样 probe 失败时，stream 阶段仍有第二次机会调 memory.search。

根因 B：Profile 里"能短则短"和"给出具体分析"自相矛盾

Profile（助手画像）里同时写着： 

## 回答风格
- 先给结论，再给理由；能短则短。           ← 这条赢了
- 用户问具体屏幕内容时，不用泛化观察索引糊弄；
  看得清就具体说，看不清就说看不清。       ← 这条被忽略

"能短则短"和"基于证据给出具体分析"在语义上是矛盾的。模型面对 15642 字符的丰富证据时， "能短则短"这条指令反而让它觉得不该展开说。修复方式是去掉矛盾： 

## 回答风格
- 先给结论，再给理由；简洁但不省略关键信息。
- 有观察数据时基于证据给出具体分析，不要泛泛而谈或一句话带过。
教训：profile 里的每一条指令都会被模型当真。当两条指令有张力时， 更短、更绝对的那条（"能短则短"）会赢过更长、更条件化的那条。 写 profile 时要主动消除指令间的矛盾，不能指望模型自己权衡。

五、借鉴 Hermes：prefix cache 冻结快照

上面都是修 bug。最后讲一个架构优化，灵感来自 Hermes Agent 的上下文管理设计。

Hermes 有一个"冻结快照"机制：会话开始时把 profile/memory 冻结成快照注入 system prompt， 中途写入只更新磁盘文件，不更新 prompt，下次会话才刷新。目的是保 prefix cache。

我的项目用 llama.cpp 本地模型，虽然没有 Anthropic 那种 API 级 cache_control 断点， 但 llama.cpp 自己有 KV cache——如果两次请求的 prompt 前缀字节级完全一致， 它会直接复用上次算好的 KV 矩阵，不重算。

问题在于，原代码每次 ask 都调 profile_packet() 重新读文件拼接。 虽然拼接结果大概率一致，但任何细微差异（空格、换行）都会导致整个前缀 KV 重算。

修复很简单——会话级冻结： 

class ChatAgent:
    def __init__(self, ...):
        self._frozen_profile_packet: str | None = None

    def _get_profile_packet(self) -> str:
        if self._frozen_profile_packet is None:
            self._frozen_profile_packet = get_profile_store().profile_packet()
        return self._frozen_profile_packet

第一次 ask 时冻结，整个会话内复用同一字符串对象。 system + profile 前缀（约 1373 字符 ≈ 700 tokens）字节级不变， llama.cpp 的 KV cache 稳定命中。连续对话时，第二、三次请求的前缀计算量从 700 tokens 降到 0， 首 token 延迟显著降低。

对桌宠式陪伴场景特别重要——用户问一句话要等 8-11 秒才出第一个字， 前缀命中能省到 4-6 秒，体感完全不同。

为什么说这是"核心创新"：Hermes 的 cache_control 面向云模型 API（按断点收费）。 本地 llama.cpp 没有 API 断点，靠的是 prompt 前缀的天然稳定性命中 KV cache。 这意味着优化方式更底层、更优雅——不是靠 API 特性，而是靠保证 prompt 结构的字节级一致性。

六、方法论复盘

  1. 不修症状，修根因。VLM 退化重复不是 max_tokens 不够，是 prompt 契约超过模型稳定输出范围。排名器失败不是算法不好，是复用了不稳定的 VLM。
  2. 当某种输出格式不稳定时，让架构不依赖它。probe 的 content 不稳定，就不让 content 承担产出职责。这不是 fallback，是职责边界。
  3. 确定性算法优先于 LLM。30-40 条候选集的排名，BM25 零失败毫秒级，LLM ranker 46% 失败 12-30 秒。小规模排名不需要语义理解。
  4. profile 指令要消除内部矛盾。"能短则短"和"给出具体分析"矛盾时，模型会选更短的那条。写 profile 要主动消除张力。
  5. prefix cache 是本地模型的关键优化点。保证 system+profile 前缀字节级一致，llama.cpp 的 KV cache 能稳定命中，首 token 延迟降低 30-60%。

指标修复前修复后

小模型不是大模型的降级版，它有自己独特的不稳定性。 架构设计的任务不是消除这些不稳定性，而是让系统的正确性不依赖于那些不稳定的部分。
