🪟 Local Window Copilot

Windows 本地陪伴式桌面伙伴 · 默认陪伴，不默认汇报

Windows 10+ Python 3.11+ FastAPI MiniCPM-V VLM Alpha · 开发中

这是什么

Local Window Copilot 是一个本地、轻量、低打扰的陪伴式桌面伙伴。 它安静地在桌面上存在，理解你的工作节奏，能被随时唤起。

它不是又一个屏幕分析器，也不是自动控制型 Agent。截图、VLM、摘要、记忆是后台感知器官， 不是产品本体。它的价值在于：

  • 🤫 平时安静陪伴 —— 看到状态变化但不急着打断
  • 💬 随时进入对话 —— 你想互动时立刻回应
  • 🔍 认真拆解分析 —— 你想分析时才调用视觉、历史和记忆
  • 🏠 完全本地运行 —— 不要求 Redis / PostgreSQL / Docker
  • 🚫 不替你操作电脑 —— 可以观察、陪伴、解释、建议，但不自动点击/输入/提交

🎭 桌宠状态

桌宠有 5 种核心状态，对应不同的工作情境，会根据你的工作节奏自然切换表情和姿态。

😴 Idle
默认陪伴，轻微呼吸
👀 Observing
注意到状态变化
🧠 Analyzing
认真分析当前屏幕
🔒 Privacy
隐私模式，停止观察
⚠️ Error
遇到问题，需要关注

✨ 核心特性

🤫
低打扰陪伴
默认沉默，不默认分析。主动发言必须满足明确条件，高质量陪伴需要会沉默。
🏠
完全本地
SQLite 存储状态与记忆，llama.cpp 本地推理，无需 Redis/PostgreSQL/Docker。
🧠
视觉理解
MiniCPM-V 视觉语言模型理解屏幕内容，但只在用户邀请时调用，不默认扫描。
🛡️
隐私优先
不替用户操作电脑，不自动点击/输入/提交。可以观察、陪伴、解释、建议、记录。
💬
陪伴对话
先接住用户的情绪和判断，再根据需要进入分析。陪伴优先于分析。
📐
克制工具层
模型可见工具只有三个：screen.look / memory.search / memory.remember，保持克制。

🎬 演示场景

以下为产品设计的典型交互场景，演示 GIF 将在正式发布后补充。

场景一：陪伴模式
用户：我觉得这个方向不对。
桌宠：我也感觉你不是在挑某个实现细节，而是在怀疑"摘要作为中心"这件事。
      我们可以先不急着改代码，把产品灵魂重新定一下。

用户：（点击桌宠"观察"按钮）

      我看到你现在在 IDE 里，打开了三个文件，光标停在 window_analysis.py。
      这个函数最近的改动引入了一个未关闭的 session，要我帮你看看吗？
场景三：低打扰主动提示
[用户在同一任务上停留 45 分钟，反复切换三个窗口]
桌宠：[轻微 observing 状态，不弹窗]
      → 仅在悬浮窗角标显示一个小提示点，用户注意到时才展开

📐 产品原则

  1. 默认不汇报屏幕 —— 屏幕分析结果默认进后台，只在用户主动邀请时展示
  2. 陪伴优先于分析 —— 先接住用户的情绪和判断，再根据需要进入分析
  3. 低打扰是能力 —— 高质量陪伴需要会沉默，主动发言必须满足明确条件
  4. 不伪装能力 —— 主链路不清楚就明确说不清楚
  5. 不替用户操作电脑 —— 可以观察、陪伴、解释、建议、记录，但不自动执行
