Agent VLM 桌面应用 复盘

为什么「看见屏幕」还不够：桌面 AI 伙伴的上下文管理踩坑

2026-07-04 · 12 min read · 宋林蔚

我做的是一个 Windows 桌面悬浮的小伙伴，能看屏幕、能聊天。有一天我随手测了一句对话，它大概是这样的：

我：要不要我帮你分析当前任务进展或下一步计划？
桌宠：你可以告诉我，我帮你分析当前任务进展或下一步计划吗？

看着像是模型在复读我上一句的提议。但它其实没坏——它只是不知道我在回应什么。 在它的视角里，我刚才那句话就是一句全新的、孤立的输入， 而我真正想说的"可以，那就帮我分析一下"这个承接关系，在它那里压根不存在。

这件事让我意识到，桌面 AI 伙伴真正难的，不是能不能截图、能不能调 VLM、能不能存历史。 难的是让模型在正确时刻拿到正确上下文，并且承认自己看不清、记不住、没证据。 这篇文章是我这个原型一路踩坑的复盘——不是为了证明我做出了什么，而是为了记录一个原型如何暴露出桌面 Agent 的核心矛盾。

图 1：早期"截图 → 摘要 → 回答"链路，看似优雅，实则在每一步都在丢信息

一、我以为截图 + 摘要就够了

原型的最初设想很直接：桌宠悬浮在桌面，实时读取当前窗口，生成一段摘要；用户提问时，把摘要塞进 prompt 让模型回答。 我当时觉得这条链路已经覆盖了"看屏幕"这件事的全部——能截、能看、能总结、能回答，齐了。

然后现实很快打脸。用户问"页面里中间那段代码具体是什么"， 摘要里只剩"这是一个代码编辑器窗口，包含若干文件"， 回答就只能复述摘要，根本说不出用户想知道的细节。 摘要越短，回答越像模板；摘要想写长，又会把上下文撑爆、拖慢响应，还把无关信息塞进模型视野。

更隐蔽的问题是，摘要一旦成为回答的唯一来源，模型就不再"看屏幕"，而是在"复述摘要"。 这两者看起来都能产出文字，但本质完全不同：前者是视觉证据驱动的判断，后者是对压缩文本的改写。 用户问细节时，摘要回答不出来；用户问感受时，摘要又显得太干巴。

我后来把摘要降级成了 RAG 索引，而不是最终证据。摘要用来回答"用户大概在哪个窗口、在做什么任务"， 但具体问题必须回看截图或局部截图。这个调整让回答质量明显改善——代价是链路变复杂了， 但这个复杂度是必须付的，因为"看见"和"理解"本来就是两件事。

踩坑要点：摘要适合做索引，不适合回答细节。"看不清"比泛泛回答更诚实。

二、VLM 能看图，但不等于理解用户此刻的问题

把摘要降级成索引之后，我开始直面第二个问题：VLM 本身的可见度。 页面在人眼看来清清楚楚，VLM 却经常说不出具体文字；文件夹窗口识别很快， 复杂网页、IDE、重叠窗口就明显变差。有时候桌宠自己的对话框还会被误当成用户正在看的目标窗口。

可能的原因列了一串：截图缩放后小字不可读、UI 层级复杂时 VLM 倾向于描述布局而不是识别内容、 目标窗口被遮挡、整图输入导致细节被稀释。每一个都好理解，但叠在一起就很折磨。

我学到的最重要的一点是：多模态模型的"视觉理解"其实同时包含 OCR、布局理解和语义理解， 而这些能力都会受分辨率、裁剪、遮挡和 prompt 影响。它不是一个"能不能看"的开关， 而是一个"在什么条件下能看清多少"的连续谱。如果要回答细节，系统应该先选图、选区域、再细看， 而不是把整张屏幕丢给模型指望它什么都认得出来。

这期间也 tempted 想加一层 OCR fallback——VLM 说不出来就用 OCR 兜底。 但我后来克制住了：OCR fallback 会掩盖主链路的问题，让"看不清"变成"看起来看清了"， 而这种假性成功比直接承认看不清更危险。主链路不行就应该暴露出来、修它，而不是用兜底糊过去。

三、对话历史不是字符串拼接

文章开头那个复读对话，本质上是上下文断线。 我说"可以"，是在回应桌宠上一轮的提议；但桌宠的 planner 只看到了孤立的"可以"二字， 没有任何承接信息，于是它只能把"可以"当成一个新任务去理解，结果就崩了。

这个问题暴露了一个很容易被忽略的事实：对话连续性不能只靠模型自己猜，系统要显式保留 turn state。 如果上一轮助手提了一个问题，而用户用"可以""继续""好的"这种短回复接话， 系统应该把上一轮的提议作为承接提示注入 planner 和 answer，而不是把短回复当成孤立输入。

我加了一个叫 dialogue bridge 的机制：当用户短回复时， 系统检测到这是承接型回复，把上一轮助手的提议拼回上下文。这不是硬编码回答， 而是恢复对话状态——告诉模型"用户在回应你刚才那句话"。 加上之后，那种复读式的失败基本消失了。

关键认知：好的上下文管理要保护"用户正在回应什么"，而不只是"用户说了什么"。

四、观察线和对话线不能混在一起

早期系统试图一边自动观察屏幕，一边回答用户问题。听上去很合理——后台一直看着，用户问什么都能答。 但实际跑起来全是问题：用户打开对话框后，自动观察可能把对话框本身当成目标窗口； 回答过程中窗口一变，上下文就漂移；观察摘要和用户问答混在一起，模型根本分不清 "后台感知到的内容"和"当前对话要回答的问题"。

图 2：观察线和对话线分离，后台建立索引但不直接打扰用户

于是我把两条线拆开了。自动观察线负责在后台默默建立屏幕索引，不直接打扰用户； 用户对话线负责承接意图、调用工具、回答问题。只有当用户点击观察按钮、 或者在对话里明确要求看屏幕时，视觉证据才会被带进对话线。 这样模型拿到的上下文永远是干净的——要么是纯对话，要么是用户明确邀请后的"对话 + 视觉"。

五、记忆管理不是加数据库

项目早期我认真考虑过引入 Redis、PostgreSQL，甚至 Docker Compose 一把梭。 听上去很"工程化"，很"可扩展"。后来我把它们全拆了。

不是因为这些技术不好，而是因为它们解决的不是我当前阶段的问题。 Redis 解决的是缓存和并发，PostgreSQL 解决的是结构化数据持久化， Docker 解决的是部署一致性——这些是规模和团队问题。 但桌宠第一阶段要面对的不是规模问题，是"该记什么、何时取出、如何注入"。 加一个 Redis 不会让模型更聪明，只会让依赖更重、启动更慢、用户体验更差。

我最终留的是 SQLite + markdown profile + runtime store。 真正重要的不是存储介质，而是把记忆分成四类、分别管理：

  • profile：用户长期偏好和助手人设，低频更新，几乎每次对话都会带。
  • working memory：最近屏幕和任务状态，高频写入，短期有效。
  • conversation memory：当前会话历史，会话结束就归档。
  • episodic memory：重要事件或项目决策，偶尔写入，长期保留。

记忆管理的核心是写入策略、检索策略和注入策略，而不是用什么数据库。 一个乱写日志的 Redis 比一个干净的 SQLite 更糟，因为它会让模型在错误时刻拿到错误的东西。

六、工具层不能变成关键词路由

原型一度滑向了一个看起来很务实的方案：用关键词决定走哪条路。 

如果用户说"看 / 页面 / 截图" -> 走视觉
如果用户说"记住"            -> 写记忆
否则                       -> 普通聊天

一开始挺好用，关键词越补越多。但很快系统就变脆了：用户换个说法就 miss， 小模型失去判断空间，用户也觉得对话死板。更糟的是，错误不会暴露在工具调用日志里， 而是表现为奇怪回答——你根本不知道是模型笨还是路由错了。

后来我把它改成了 Hermes-like 的工具层思路（受到启发，不是照搬）： 让 LLM 自己在少量工具里选择，工具执行后返回证据，再由 answer model 基于证据组织回答。

图 3：上下文分层，稳定部分靠前保护 KV cache，动态部分作为工具结果注入

当前暴露给模型的工具只有四个：

  • screen.look(question)：看当前屏幕或用户图片。
  • memory.search(query)：查 profile、历史对话、近期目标。
  • memory.remember(note)：仅用户明确要求时写入。
  • none：普通陪伴和讨论，不调工具。

后端 provider 可以很复杂——要管截图索引、窗口 metadata、局部裁剪、VLM 分层 messages—— 但模型可见的工具要少、要稳定、要可审计。 Hermes-like 思想给我的启发不是"工具越多越 Agent"，而是"工具边界要清楚、可审计、可调试"。 工具调用日志也应该暴露给高级调试，否则系统会越来越像黑盒，出了问题根本定位不到。

关于 KV cache：稳定的 system prompt 只放协议、边界、回答原则，是为了保护可缓存 prefix。 动态上下文（当前屏幕、历史截图、记忆检索结果）作为工具结果注入，而不是揉成一段巨大的 prompt。 KV cache 不是记忆，只是性能优化——为了缓存，系统提示词要稳；为了回答质量，动态上下文要分层。

七、产品方向：从"屏幕摘要器"转向"陪伴式桌宠"

上面这些技术调整累积起来，把我对产品的理解也改变了。

原型早期我满脑子都是"它能看屏幕、它能摘要、它能回答"，越强调摘要，桌宠就越像一个机械的屏幕分析器—— 每隔几秒弹一段"当前你在看 XX 窗口"，用户不问它也在汇报。这不是陪伴，这是骚扰。

后来我把产品中心从"摘要"切换到"陪伴"：

默认陪伴，不默认汇报。
默认沉默，不默认分析。
用户邀请时，才认真看、认真想、认真回应。

截图、VLM、摘要、记忆仍然重要，但它们是后台感知器官，不是产品本体。 产品本体是一个在桌面上安静存在、理解工作节奏、能被随时唤起的小伙伴。 它平时不打扰，用户想互动时立刻进入对话，用户想分析时再认真拆解。

八、失败给出的架构原则

回头看这一路，真正有价值的不是任何一个具体修复，而是几次失败之后慢慢沉淀下来的几条判断：

  1. 摘要是索引，不是答案。摘要用来定位"用户大概在哪、在做什么"，细节问题必须回看证据。
  2. 截图是证据，不是上下文全部。视觉信息要按需取用，而不是无差别塞进 prompt。
  3. 记忆是选择，不是日志。该记什么、何时取出、如何注入，比用什么数据库更重要。
  4. 工具是协议，不是关键词分支。让模型在少量稳定工具里选择，比写一长串 if-else 更可持续。
  5. 陪伴式 Agent 的价值在于"接住用户"，不是"炫耀能力"。低打扰是一种能力，会沉默比会说话更难。

下一步计划

接下来我想做三件事：第一，让模型通过少量工具自由选择上下文，摘要只做索引不进回答； 第二，记忆写入必须可解释——什么时候写、写了什么、为什么写，都要能查得到； 第三，把上下文使用率和交互轨迹做成可调试的视图，让自己能在出问题时第一时间看到 "模型当时到底拿到了什么"。

我不打算把它包装成一个"已经成熟的 Agent"。它更准确的状态是—— 一个正在逼近正确问题的原型。很多失败暴露了桌面 AI 伙伴最难的工程部分， 而目前的价值，在于形成了还算清晰的架构判断。

后来我意识到，一个桌面 AI 伙伴的核心不是每秒总结屏幕， 而是在用户需要它的时候，知道该看哪里、该记起什么、该承认什么不知道， 以及该怎样像一个在场的伙伴那样回应。
